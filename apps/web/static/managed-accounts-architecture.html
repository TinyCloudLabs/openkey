Architecture brief Managed accounts v1

App-managed by default. User‑owned by choice.

OpenKey now gives applications a policy-bounded way to provision and manage user accounts—without making the application the permanent owner. A user can transfer the unchanged account to personal OpenKey custody at any time.

  • TenancyOrganizations isolate apps, credentials, subjects, keys, and quotas.
  • IdentityThe user establishes an independent OpenKey passkey before management activates.
  • ContinuityAddress, DID, TinyCloud spaces, and encrypted data survive eject unchanged.

The product promise

The new model separates a useful managed experience from permanent application ownership.

An app can give every user an embedded OpenKey and TinyCloud account. The app manages it by default. The user can leave without changing their address, DID, spaces, or data. “Personal custody” means the unchanged private key remains sealed inside OpenKey’s TEE. It is not private-key export or external self-custody.

What changed

The previous user-and-client shape has become an explicit custody control plane with durable tenant boundaries.

Free, Pro, and Enterprise resolve versioned quotas without ever gating a user’s right to eject.

The system boundary

OpenKey is the custody and capability control plane. Tenant applications operate through organization credentials and bounded delegations; TinyCloud nodes remain protocol-generic.

Organization identity is derived from the authenticated server credential. It is never trusted from a tenant-supplied request field.

Global user

One OpenKey principal may use several apps, but credentials and recovery remain private to the user.

Tenant-specific account

Each organization relationship creates a distinct managed key. Matching email or external IDs cannot merge custody.

Typed tenant access

The broker may request approved TinyCloud capabilities within policy and plan ceilings. There is no tenant raw-sign endpoint.

Stable key identity

The key’s sealingContext, address, and account identity do not change when custody changes.

Registration now starts with the app—and belongs to the user

The tenant initiates the flow, but OpenKey owns the credential ceremony and will not activate management until the user has an independent passkey.

Custody is explicit, versioned, and terminal

Every sensitive operation resolves the authenticated actor, current custody head, lifecycle state, policy, plan, and expected epoch immediately before the TEE key is used.

USER_OWNED is a custody result. TinyCloud revocation convergence is tracked separately so the UI never confuses committed personal custody with network propagation.

Eject is unilateral; revocation converges afterward

The application cannot approve, delay, cancel, or reverse eject. The synchronous transaction removes OpenKey broker authority first; the worker then confirms protocol revocation on every known TinyCloud node.

Honest security boundary

The demo stores signed, hash-chained custody events in PostgreSQL. A production rollout still needs an external monotonic witness and old-image denial before OpenKey can claim rollback-resistant custody ordering.

Plans control service levels—not the right to leave

Entitlements answer how much service an organization may consume. They cannot broaden a cryptographic capability, delete user data, or block authentication, reads, export, or eject.

The invariants enforced in code and SQL

The service layer is not the only line of defense. PostgreSQL constraints and triggers protect custody coherence when writes bypass normal routes.

  • One owner, one active custodianA managed account has one stable owner and one custody head at its exact current epoch.
  • Immutable key identityManaged key classification, address, public key, and sealing context cannot be rewritten after binding.
  • One-way custodyOrganization custody may transition to user custody once. Returning a user-owned account to tenant management is rejected.
  • Central authorizationManaged key routes cannot authorize through an ad hoc key-and-user lookup; state and epoch are rechecked under lock.
  • Tenant isolationIntent creators, OAuth clients, managed accounts, webhook endpoints, and deliveries must share the same organization.
  • Bounded delegationRequested capabilities must fit both the immutable OpenKey ceiling and the account’s versioned policy.
  • Fresh owner proofEject requires a recent server-recorded passkey ceremony with user verification, not a client assertion.
  • Durable idempotencyRegistration, eject, lifecycle delivery, and revocation work are replay-safe and preserve completed results.
